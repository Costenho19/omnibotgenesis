# OMNIX — Data Retention Policy

**Version**: 1.0  
**Date**: February 27, 2026  
**Author**: Harold Nunes, OMNIX  
**Applies to**: External B2B clients using the Governance API (ADR-028)

---

## Purpose

This policy defines how long OMNIX retains data generated by external B2B clients through the Governance API (`POST /api/governance/evaluate`), and the technical controls in place to protect client data at rest.

---

## Scope

| Data Type | Table | Covered by This Policy |
|-----------|-------|----------------------|
| Governance receipts | `decision_receipts` | Yes |
| Shadow trade events | `shadow_trade_events` | Yes |
| Internal bot decisions | `decision_receipts` (client_id = NULL) | No — internal use |

---

## Retention Period

**Default**: 12 months from receipt creation date.

Each governance receipt stores a `retention_until` date, calculated at ingestion time:

```
retention_until = created_at + 365 days
```

This value is set automatically by `DecisionReceiptEngine.store_receipt()` at the application layer.

**Configurable per contract**: The `retention_until` column may be extended for enterprise clients by contract. Shorter periods (e.g., 90 days for compliance-sensitive verticals) may also be negotiated.

---

## Data at Rest — Encryption

Client signal payloads are encrypted at the application layer before storage.

**Mechanism**: Fernet symmetric encryption (AES-128-CBC + HMAC-SHA256)  
**Key management**: `PAYLOAD_ENCRYPTION_KEY` environment variable — stored exclusively in Railway Variables, never in the codebase  
**Scope**: The `encrypted_payload` column in `decision_receipts` stores the encrypted JSON of client signals  
**Receipts themselves** (receipt_id, decision, hash, PQC signature) are stored in plaintext — required for cryptographic verification integrity

**Statement for due diligence**:

> "Client signal data submitted through the Governance API is encrypted at rest at the application layer using AES-128-CBC with HMAC-SHA256 integrity verification. Encryption keys are managed exclusively via environment secrets, not stored in source code."

---

## Multi-Tenant Isolation

Each governance request is tagged with a `client_id` (from the `X-Client-ID` request header). This field is stored in:
- `decision_receipts.client_id`
- `shadow_trade_events.client_id`

This enables per-client data queries and future isolation policies. Client data is not cross-exposed — queries for one client_id do not surface another client's data.

---

## Deletion Process

Data cleanup is performed manually or via a scheduled Railway job using:

```sql
-- Preview what would be deleted
SELECT COUNT(*), client_id
FROM decision_receipts
WHERE retention_until < CURRENT_DATE
GROUP BY client_id;

-- Execute deletion (run after preview confirmation)
DELETE FROM decision_receipts
WHERE retention_until < CURRENT_DATE;
```

**Frequency**: Reviewed quarterly or upon client contract termination.  
**Responsibility**: OMNIX infrastructure administrator (Harold Nunes).

---

## Client Rights

Upon contract termination, a client may request:
1. **Export**: All receipts tagged with their `client_id` exported as JSON
2. **Deletion**: All records with their `client_id` deleted ahead of the standard retention schedule

These requests are fulfilled within 30 days of written request.
