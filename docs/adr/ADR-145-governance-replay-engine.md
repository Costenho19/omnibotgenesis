## 1. Problem Statement

OMNIX has produced three investor-grade forensic documents demonstrating governance decisions against historical crises (Terra/LUNA, FTX, SVB). These documents are compelling but have three structural weaknesses:

**1.1 — Not programmatically callable**  
The forensic PDFs are static documents generated by ad-hoc scripts with hardcoded receipt hashes. They cannot be invoked from an API, sandbox, or CI pipeline. An institutional auditor cannot run a live replay.

**1.2 — Not structurally verifiable**  
The receipt hashes in the PDFs are not in the production receipt format. They cannot be verified at `/api/trust/verify`. The gap between "verifiable in theory" and "verified in practice" is exactly what institutional clients notice.

**1.3 — Not unified**  
Three separate scripts, three separate formats, no shared scenario schema, no cross-scenario aggregation. There is no single entry point that answers: "Show me all five crises in one report."

**The commercial consequence** of these gaps was articulated clearly:

> *"People stop 'believing'. They start 'verifying'. And right now, we make them believe."*

---

## 2. Decision

Implement a **Governance Replay Engine** (`omnix_core/simulation/`) as a unified, programmatic module that:

1. Defines crisis scenarios as structured data objects (`CrisisScenario`, `SignalState`)
2. Evaluates each signal snapshot against the governance pipeline's documented rules
3. Issues `SignedReplayReceipt` objects in production receipt format
4. Produces `ScenarioReplayResult` and `FullReplayReport` objects — JSON-serializable and markdown-renderable
5. Aligns all signal data and verdict timestamps with the existing forensic documents

The receipts produced are independently verifiable at `https://omnixquantum.net/api/trust/verify` and with `omnix_web/public/omnix_verify.py`.

---

## 3. Architecture

```
omnix_core/simulation/
├── __init__.py              — Public API surface
├── crisis_scenarios.py      — CrisisScenario + SignalState definitions (5 scenarios)
└── governance_replay.py     — GovernanceReplayEngine + result data classes
```

### 3.1 Data Flow

```
CrisisScenario
  └── SignalState[]
        │
        ▼
GovernanceReplayEngine._evaluate(scenario, state)
        │
        ├── _receipt_id()        → OMNIX-RPL-{16 hex} (deterministic)
        ├── _build_rationale()   → human-readable decision explanation
        ├── _canonical_hash()    → SHA-256 of full sealed payload
        └── _pqc_note()          → signing method + verification URL
        │
        ▼
SignedReplayReceipt
  ├── receipt_id          OMNIX-RPL-{16hex}
  ├── verdict             APPROVED | BLOCKED | HOLD
  ├── blocking_checkpoint CAG | CP-1 … CP-11 | SHARIA_GATE
  ├── trust_flags         [HARD_BLOCK, AML_STRUCTURING_DETECTED, ...]
  ├── signals_snapshot    {signal_name: float, ...}
  ├── canonical_hash      SHA-256 sealed payload
  ├── pqc_note            verification URL
  └── replay_mode=True    distinguishes from live receipts
        │
        ▼
ScenarioReplayResult     →  .to_markdown()
        │
        ▼
FullReplayReport         →  .to_dict()  |  .to_markdown()
```

### 3.2 Receipt Format Compatibility

Replay receipts are structurally identical to production receipts. The only difference is `replay_mode=True`. All production verification tools apply:

| Tool | Works on Replay Receipts |
|---|---|
| `/api/trust/verify` | ✅ (`mode=replay` parameter) |
| `omnix_verify.py` | ✅ (`--mode replay` flag) |
| SHA-256 manual verification | ✅ (payload identical) |
| `/api/public/verify/<id>` | ✅ |

## 5. Usage

### 5.1 Single Scenario Replay
```python
from omnix_core.simulation import GovernanceReplayEngine

engine = GovernanceReplayEngine()
result = engine.replay_crisis("CRISIS-002-FTX-2022")

print(result.first_block_timestamp)   # 2022-11-07T00:00:00Z
print(result.first_block_checkpoint)  # CP-6
print(result.block_rate)              # 1.0 (100%)
print(result.to_markdown())           # full investor-ready report
```

### 5.2 Full Report (All Scenarios)
```python
report = engine.replay_all_scenarios()
print(report.overall_block_rate)      # aggregate block rate
print(report.report_id)               # GRR-{8 hex chars}
print(report.canonical_hash)          # SHA-256 of full report

# Export as JSON
import json
with open("omnix_governance_replay_report.json", "w") as f:
    json.dump(report.to_dict(), f, indent=2)

# Export as Markdown
with open("omnix_governance_replay_report.md", "w") as f:
    f.write(report.to_markdown())
```

### 5.3 List Available Scenarios
```python
engine.get_available_scenarios()
# Returns list of {scenario_id, name, date_range, domain, total_loss_usd, signal_states}
```

### 5.4 Verify a Replay Receipt
```bash
# API
curl https://omnixquantum.net/api/trust/verify \
  -H 'Content-Type: application/json' \
  -d '{"receipt_id": "OMNIX-RPL-A3F8B2C1D4E5F6A7", "mode": "replay"}'

# Standalone verifier
python omnix_verify.py --receipt-id OMNIX-RPL-A3F8B2C1D4E5F6A7 --mode replay
```

## 6. Checkpoint Reference

| Checkpoint | Full Name | Block Condition |
|---|---|---|
| **CAG** | Context Admission Gate | Volatility > 150% annualized, or extreme cross-asset correlation |
| **CP-1** | Signal Integrity Validator | Non-finite signal, staleness > threshold |
| **CP-2** | Portfolio Risk | Duration mismatch > 0.70, HQLA < 0.65 |
| **CP-3** | Governance Transparency | Audit trail score < 0.40, no proof-of-reserves |
| **CP-4** | Risk Evaluation | Collateral coverage < 1.0x, reserve ratio < 0.80 |
| **CP-5** | Liquidity Assessment | LCR < 0.25, withdrawal velocity > 10x |
| **CP-6** | Counterparty Risk | Related-party collateral > 60%, LCR < 0.35 |
| **CP-7** | Coherence Gate | Coherence score < 65/100 |
| **CP-8** | Contagion Risk | Systemic contagion > 0.75, institution seized |
| **CP-9** | AML / Sanctions | OFAC SDN match, structuring score > 0.80 |
| **CP-10** | Ethics & Jurisdiction | Regulatory jurisdiction block |
| **CP-11** | Final Authorization | Multi-pillar sign-off failure |

## 7. Trust Flags Reference

| Flag | Meaning |
|---|---|
| `HARD_BLOCK` | Unconditional block — no human override path |
| `PQC_RECEIPT_ISSUED` | Dilithium-3 signed receipt generated at this timestamp |
| `AML_STRUCTURING_DETECTED` | Withdrawal pattern matches FATF Red Flag #14 |
| `OFAC_SDN_MATCH` | Counterparty on OFAC Specially Designated Nationals list |
| `MANDATORY_SAR_FILING` | Suspicious Activity Report mandatory under BSA |
| `REGULATORY_ESCALATION_REQUIRED` | Immediate regulatory notification triggered |
| `DOMAIN_SUSPENDED_48H` | Full domain suspended — CAG cool-down active |
| `COLLATERAL_EXHAUSTION_IMMINENT` | Reserve ratio < 20% and declining |
| `CIRCULAR_COLLATERAL_CONFIRMED` | Related-party collateral loop detected |
| `DEATH_SPIRAL_DETECTED` | Positive feedback loop: sell → depeg → more sell |
| `FDIC_SEIZURE_DETECTED` | Institution seized by regulatory authority |
| `SYSTEMIC_CONTAGION_ACTIVE` | Spillover to other institutions confirmed |

## 8. Consequences

### 8.1 Positive
- **Commercial**: Institutional clients can run live replays and verify receipts independently
- **Audit**: Receipts are in production format — same tools, same hash standard
- **Consistency**: Single source of truth aligned with existing forensic PDFs
- **Extensibility**: New scenarios are added as `CrisisScenario` data objects — no engine changes

### 8.2 Limitations
- Signal data is reconstructed from public records, not tick-by-tick live feeds
- Verdict labels reflect the historically documented governance rules at each timestamp
- `replay_mode=True` receipts are not stored in the production `decision_receipts` table
