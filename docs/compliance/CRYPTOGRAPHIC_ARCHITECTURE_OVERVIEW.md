**Classification**: Institutional — Investor & Partner Distribution

## Strategic Context

Automated decision systems handling capital allocation, risk governance, and regulatory compliance will face a fundamental cryptographic challenge: the arrival of fault-tolerant quantum computers capable of breaking current public-key cryptography.

OMNIX addresses this proactively. Post-quantum cryptography is not a roadmap item — it is operational infrastructure, active since November 2025.

This is a strategic differentiator: OMNIX decisions are signed with algorithms designed to remain secure in a post-quantum world.

## Cryptographic Capabilities

### Decision Signing — Configurable Assurance Tiers (CRYSTALS-Dilithium)

Every automated governance decision is cryptographically signed before execution using CRYSTALS-Dilithium, a NIST-standardized digital signature algorithm family (ML-DSA).

The signing level is **deployment-context configurable**. Security posture scales with the operational environment — no architectural rewrite required.

| Configuration | Algorithm | NIST Variant | Security | Target Use Case |
|---------------|-----------|--------------|----------|-----------------|
| **Enterprise Baseline** (current production) | Dilithium-3 | ML-DSA-65 | ~192-bit classical equivalent | Capital governance, institutional deployments, B2B governance infrastructure |
| **High-Assurance** (configurable) | Dilithium-5 | ML-DSA-87 | ~256-bit classical equivalent | National-grade deployments, regulated environments requiring maximum assurance |

**Why Level 3 for the enterprise baseline:**

Level selection is threat-model driven, not arbitrary. NIST Level 3 (~AES-192 equivalent) is the designed target for capital-sensitive, institutional environments — it is not a "minimum viable" security level. Level 5 (~AES-256 equivalent) is appropriate for state-secrecy or national-grade contexts where signature overhead is acceptable. The governance architecture is parameterizable between tiers via a single deployment-environment configuration.

**What this provides:**

- **Non-repudiation** — Every decision has a verifiable, tamper-evident signature proving it was generated by the governance engine
- **Auditability** — Regulators and auditors can independently verify that decisions were not modified after generation
- **Accountability** — The signature chain creates an immutable record linking each decision to the exact governance state that produced it
- **Context-appropriate assurance** — Cryptographic level matches the operational threat model, not a marketing ceiling

**Operational integration:**

All real trading orders include:
- `pqc_signed`: Confirmation of cryptographic signing
- `pqc_algorithm`: Algorithm identifier (e.g., Dilithium-3 / ML-DSA-65)
- Signature payload linked to order parameters (symbol, amount, price, timestamp, user)

### Key Encapsulation — Not Data Encryption (CRYSTALS-Kyber-768)

Secure key exchange uses CRYSTALS-Kyber, a NIST-standardized key encapsulation mechanism (ML-KEM-768), providing quantum-resistant protection for:

- API key transmission and storage
- Inter-service shared secret establishment
- Quantum-resistant key exchange

**Important distinction for compliance and due diligence:** Kyber-768 is a Key Encapsulation Mechanism (KEM) — it establishes a shared secret between parties. Actual payload encryption uses symmetric AES/Fernet. This is cryptographic best practice: PQC-KEM for key exchange + symmetric cipher for bulk data. Describing Kyber-768 as a "data encryption" algorithm would be technically incorrect.

---

### Quantum Resistance

Both algorithms are designed to resist attacks from both classical and quantum computers, protecting against:
- Classical brute-force and cryptanalytic attacks
- Quantum attacks via Shor's algorithm (factoring/discrete log)
- Quantum speedup via Grover's algorithm (symmetric key search)

Both operate at strong security levels aligned with NIST standardized guidance.

## Graceful Degradation

The PQC module is designed for resilience:
- If the cryptographic library is unavailable, the system logs a warning and continues operation
- `PQC_AVAILABLE` flag allows runtime detection
- No hard dependency — the governance engine operates with or without PQC, but PQC is always active in production
- Invalid tier configuration fails closed to Level 3 — the system never silently degrades to unsigned operation
