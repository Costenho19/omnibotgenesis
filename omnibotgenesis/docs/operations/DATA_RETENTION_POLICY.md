# OMNIX — Data Retention Policy

**Version**: 2.0  
**Date**: February 27, 2026  
**Author**: Harold Nunes, OMNIX  
**Applies to**: External B2B clients using the Governance API (ADR-028)

---

## Purpose

This policy defines how long OMNIX retains data generated by external B2B clients through the Governance API (`POST /api/governance/evaluate`), and the technical controls in place to protect and isolate client data at rest.

---

## Scope

| Data Type | Table | Covered by This Policy |
|-----------|-------|----------------------|
| Governance receipts | `decision_receipts` | Yes |
| Shadow trade events | `shadow_trade_events` | Yes |
| B2B client registry | `b2b_clients` | Yes (access control) |
| Internal bot decisions | `decision_receipts` (client_id = NULL) | No — internal use |

---

## Retention Period

**Default**: 12 months from receipt creation date.

Each governance receipt stores a `retention_until` date, calculated at ingestion time:

```
retention_until = created_at + 365 days
```

This value is set automatically by `DecisionReceiptEngine.store_receipt()` at the application layer.

**Configurable per contract**: The `retention_until` column may be extended for enterprise clients by contract. Shorter periods (e.g., 90 days for compliance-sensitive verticals) may also be negotiated.

---

## Data at Rest — Encryption

Client signal payloads are encrypted at the application layer before storage.

**Mechanism**: Fernet symmetric encryption (AES-128-CBC + HMAC-SHA256)  
**Key management**: `PAYLOAD_ENCRYPTION_KEY` environment variable — stored exclusively in Railway Variables, never in the codebase  
**Scope**: The `encrypted_payload` column in `decision_receipts` stores the encrypted JSON of client signals  
**Receipts themselves** (receipt_id, decision, hash, PQC signature) are stored in plaintext — required for cryptographic verification integrity

**Statement for due diligence**:

> "Client signal data submitted through the Governance API is encrypted at rest at the application layer using AES-128-CBC with HMAC-SHA256 integrity verification. Encryption keys are managed exclusively via environment secrets, not stored in source code."

## Access Control — RBAC

Client authentication and isolation is enforced through a Role-Based Access Control system backed by the `b2b_clients` table.

### Architecture

| Component | Description |
|-----------|-------------|
| `b2b_clients` table | Stores client registry: client_id, api_key_hash (SHA-256), name, email, role, is_active |
| API key storage | Plaintext keys are **never stored** — only SHA-256 hash is persisted |
| Authentication | `X-API-Key` header → SHA-256 → lookup in `b2b_clients WHERE is_active = TRUE` |
| client_id source | Always from DB (authenticated record), **never from the request header** — prevents spoofing |
| Role enforcement | `standard` role: evaluate + view own receipts. `admin` role: + manage clients |

### Roles

| Role | Permissions |
|------|-------------|
| `standard` | `POST /api/governance/evaluate`, `GET /api/governance/receipts` (own only) |
| `admin` | All standard + `GET/POST/DELETE /api/governance/admin/clients`, key rotation |

### Isolation Guarantee

A client authenticated with role `standard` can only retrieve their own receipts:

```sql
SELECT * FROM decision_receipts WHERE client_id = [authenticated_client_id]
```

No client can access another client's receipts, even with a valid API key.

### Client Lifecycle

| Action | Endpoint | Auth |
|--------|----------|------|
| Create client | `POST /api/governance/admin/clients` | admin |
| List clients | `GET /api/governance/admin/clients` | admin |
| Deactivate client | `DELETE /api/governance/admin/clients/<id>` | admin |
| Reactivate client | `POST /api/governance/admin/clients/<id>/reactivate` | admin |
| Rotate API key | `POST /api/governance/admin/clients/<id>/rotate` | admin |

Deactivation is instant — the client's next request returns 401 without code changes or deployment.

**Statement for due diligence**:

> "Each B2B client is issued a unique cryptographic API key. Client identities are validated server-side on every request. A client can only access their own governance receipts. Client access can be revoked instantly without system restart."

---

## Multi-Tenant Isolation

Each governance request is tagged with the authenticated `client_id` from the `b2b_clients` registry. This field is stored in:
- `decision_receipts.client_id`
- `shadow_trade_events.client_id`

Client data is not cross-exposed — queries for one client_id do not surface another client's data.

---

## Deletion Process

Data cleanup is performed manually or via a scheduled Railway job using:

```sql
-- Preview what would be deleted
SELECT COUNT(*), client_id
FROM decision_receipts
WHERE retention_until < CURRENT_DATE
GROUP BY client_id;

-- Execute deletion (run after preview confirmation)
DELETE FROM decision_receipts
WHERE retention_until < CURRENT_DATE;
```

**Frequency**: Reviewed quarterly or upon client contract termination.  
**Responsibility**: OMNIX infrastructure administrator (Harold Nunes).

---

## Client Rights

Upon contract termination, a client may request:
1. **Export**: All receipts tagged with their `client_id` exported as JSON
2. **Deletion**: All records with their `client_id` deleted ahead of the standard retention schedule

These requests are fulfilled within 30 days of written request.
